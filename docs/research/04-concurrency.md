# Phase 1.4: Concurrency — Comparative Analysis

This document covers the fourth research topic from the Cobalt roadmap: async/await models, threading, data-race prevention, and concurrency patterns in C# and Rust, and what these mechanics imply for a language that combines both models on the .NET runtime.

This document intentionally does not revisit ownership rules, borrowing, or memory management — those are covered in documents 01, 02, and 03. The focus here is on the concurrency machinery itself: how each language represents and executes concurrent work, how each prevents (or fails to prevent) data races, and how those models interact with the .NET runtime.

---

## 1. Async/Await Models

### Rust: Poll-Based Futures

Rust's async model is built on the `Future` trait, defined in `std::future`:

```rust
pub trait Future {
    type Output;
    fn poll(self: Pin<&mut Self>, cx: &mut Context<'_>) -> Poll<Self::Output>;
}
```

The `poll` method attempts to resolve the future to a final value without blocking. It returns `Poll::Ready(value)` when the computation is complete, or `Poll::Pending` when it cannot yet make progress. The `Context` parameter carries a `Waker` — a handle that the future stores when it returns `Pending` so that the runtime can re-poll it when the underlying I/O or timer signals readiness.

**Lazy evaluation.** A Rust future does nothing until something polls it. Calling an `async fn` returns a future value but does not begin any work. This is fundamentally different from C#'s model. The implication is that constructing a future is cheap and side-effect-free, but it also means that futures that are never awaited never execute at all. The compiler warns about unused futures for this reason.

**Async fn desugaring.** When the compiler encounters `async fn foo() -> T`, it desugars the function into one that returns `impl Future<Output = T>`. The function body becomes a state machine — a compiler-generated enum whose variants correspond to the suspension points (each `.await`). Each variant stores the local variables that are live across that suspend point. The compiler advances the state machine each time `poll` is called, executing the next segment of the function body until it either completes or hits the next `.await`.

**Pin and self-referential state machines.** The state machine generated by an async function may contain self-referential data: a local variable in one state may hold a reference to another local variable. If the state machine were moved in memory after such a reference was created, the reference would dangle. `Pin<&mut Self>` in the `poll` signature prevents this. `Pin` is a wrapper around a pointer that prevents the pointee from being moved. Types that do not contain self-referential data implement the `Unpin` marker trait, which makes `Pin` transparent — `Pin<&mut T>` behaves identically to `&mut T` for `Unpin` types. The `PhantomPinned` marker type opts a type out of `Unpin`, requiring explicit pinning.

Pin is a library construct, not a compiler primitive. It works by restricting safe access to the inner pointer: safe code cannot obtain `&mut T` from `Pin<&mut T>` when `T: !Unpin`, which prevents calling `mem::swap` or `mem::replace` on the pinned value. The `unsafe` function `Pin::get_unchecked_mut` exists for cases where the caller can uphold the pinning invariant manually.

**No built-in runtime.** The Rust standard library defines `Future`, `Poll`, `Waker`, `Pin`, and a minimal executor in `std::task`, but it does not include an async runtime. Polling futures, managing I/O event loops, and scheduling tasks are the responsibility of external crates. The dominant runtimes are:

- **tokio** — the most widely used runtime, providing a multi-threaded work-stealing scheduler, timers, I/O drivers (epoll/kqueue/IOCP), and a rich ecosystem of async utilities (channels, synchronization primitives, file I/O, networking).
- **async-std** — an alternative runtime that mirrors the `std` library's API surface with async equivalents. Less widely adopted than tokio but architecturally similar.
- **smol** — a lightweight runtime emphasizing simplicity and small dependency footprint.

This separation means that a Rust binary's async behavior depends on which runtime it links. Library crates that are runtime-agnostic use only `Future` and `Poll`; those that need timers or I/O must either depend on a specific runtime or use abstraction traits.

**The Waker mechanism.** When a future returns `Poll::Pending`, it must arrange to be woken when progress is possible. The `Waker` provided via `Context` is a type-erased callback. The future clones the `Waker` and passes it to whatever subsystem it is waiting on (an I/O driver, a timer, a channel). When that subsystem detects readiness, it calls `Waker::wake()`, which signals the runtime to re-poll the future. The contract requires that only the `Waker` from the most recent `poll` call be used — a future that stores an old `Waker` and ignores a new one may fail to be woken.

**Cancellation by dropping.** Rust has no `CancellationToken`. A future is cancelled by dropping it. When a future is dropped, its destructor runs, cleaning up any resources the state machine holds. This is deterministic and immediate — there is no cooperative check, no exception, and no polling. The partially-completed computation simply ceases to exist. Any resources acquired by the future (file handles, network connections, locks) are released through their own `Drop` implementations.

This model is elegant but has a subtle consequence: if a future is dropped between two `.await` points, the code after the second `.await` never executes. This means that cleanup logic placed after an `.await` may be skipped on cancellation. Robust async Rust code must account for cancellation at every `.await` point — any `.await` might be the last thing the future does. The idiomatic solution is to rely on RAII guards for resource cleanup rather than sequential cleanup code.

### C#: Task-Based Async

C#'s async model is built on `Task` and `Task<T>` (and, for performance-critical paths, `ValueTask` and `ValueTask<T>`). The compiler transforms an `async` method into a state machine that implements `IAsyncStateMachine`, similar in structure to Rust's desugaring but with several key differences in semantics.

**Hot start (eager execution).** When an `async` method is called, it begins executing immediately on the caller's thread — synchronously — until it hits its first `await` of an incomplete operation. At that point, the method suspends, the state machine captures its state, and control returns to the caller. The returned `Task<T>` represents the ongoing work. This is the opposite of Rust's lazy model: in C#, merely calling an async method has side effects. The work is already underway.

**The state machine transformation.** The C# compiler generates a struct (for `async Task` methods, promoted to the heap when the first suspension occurs) implementing `IAsyncStateMachine`. The struct has a `MoveNext()` method containing a switch statement over the current state, mirroring the original method's control flow. Each `await` expression becomes a state transition. The builder (`AsyncTaskMethodBuilder<T>`) manages the `Task` object, sets the result when the state machine completes, and hooks up continuations via awaiters.

**Task<T> vs ValueTask<T>.** `Task<T>` is a reference type allocated on the heap. Every call to an async method that actually suspends allocates a `Task<T>` to represent the pending operation. `ValueTask<T>` is a value type (struct) that can wrap either a `T` (for synchronously completed operations) or an `IValueTaskSource<T>` (for async completion backed by a pooled object). `ValueTask<T>` avoids the heap allocation for the common case where the result is available synchronously. The tradeoff: `ValueTask<T>` instances may only be awaited once and cannot be cached or used with combinators like `Task.WhenAll`. They are a performance optimization for hot paths, not a general replacement for `Task<T>`.

**SynchronizationContext and ConfigureAwait.** After an `await` completes, the continuation (the rest of the method after the `await`) must resume somewhere. By default, C# captures the current `SynchronizationContext` before suspending and posts the continuation back to that context. In a UI application (WPF, WinForms), this means the continuation resumes on the UI thread. In ASP.NET (pre-Core), it resumes on a request-context thread.

`SynchronizationContext` is an abstract base class with `Post` (asynchronous dispatch) and `Send` (synchronous dispatch) methods. Subclasses override these to implement context-specific scheduling. The default `SynchronizationContext` posts work to the thread pool.

`ConfigureAwait(false)` tells the awaiter not to capture the `SynchronizationContext`, allowing the continuation to resume on any thread pool thread. This is standard practice in library code to avoid unnecessary context switches and deadlocks. The need to sprinkle `ConfigureAwait(false)` throughout library code is widely considered an ergonomic pain point of C#'s async model. ASP.NET Core removed its `SynchronizationContext` entirely, making `ConfigureAwait(false)` unnecessary in that environment — but library code that might run in other contexts still needs it.

**CancellationToken.** C# uses cooperative cancellation via `CancellationTokenSource` and `CancellationToken`. A `CancellationTokenSource` creates tokens; calling `Cancel()` on the source signals all tokens derived from it. Async methods accept a `CancellationToken` parameter and periodically check `token.IsCancellationRequested` or call `token.ThrowIfCancellationRequested()`, which throws `OperationCanceledException`. The `Task` infrastructure recognizes this exception and transitions the task to the `Canceled` state.

Cancellation is explicit and cooperative — the async method must check the token. If it does not, cancellation has no effect. Tokens can be composed: `CancellationTokenSource.CreateLinkedTokenSource` creates a child token that is cancelled when any parent is cancelled. Callbacks can be registered via `token.Register()` for cleanup when cancellation is requested.

**Async streams (IAsyncEnumerable<T>).** C# 8 introduced `IAsyncEnumerable<T>` and `await foreach`, enabling asynchronous iteration. An async iterator method uses `yield return` inside an `async` method returning `IAsyncEnumerable<T>`. The consumer uses `await foreach` to pull items one at a time, awaiting each `MoveNextAsync()` call. Cancellation is supported via the `[EnumeratorCancellation]` attribute on a `CancellationToken` parameter. `IAsyncEnumerable<T>` is built on `ValueTask<bool>` for `MoveNextAsync`, minimizing allocations during iteration.

Rust has no direct equivalent in the standard library, though the `Stream` trait in `futures-core` serves a similar role. `Stream` is essentially `Future` generalized to yield multiple values — its `poll_next` method returns `Poll<Option<Item>>`. As of 2025, `Stream` has not been stabilized in `std`, and async iteration in Rust requires third-party crates.

### Key Differences

**Lazy vs eager execution.** This is the most fundamental difference. Rust futures are inert until polled; C# tasks begin executing immediately. In Rust, creating a future is a pure value construction. In C#, calling an async method runs code. This affects composition: in Rust, you can construct a collection of futures and decide later how to execute them (sequentially, concurrently, with a timeout). In C#, by the time you have a collection of `Task` objects, all of them are already running.

**Built-in vs external runtime.** C#'s async infrastructure is part of the runtime: the thread pool, `Task` scheduler, and `SynchronizationContext` are always available. Rust delegates all scheduling to user-chosen runtimes. This gives Rust more flexibility (embedded systems can use a single-threaded executor with no heap allocation) but creates ecosystem fragmentation (tokio-specific code does not compose with async-std code without adapters).

**Cancellation models.** Rust's drop-based cancellation is automatic, deterministic, and requires no cooperation from the future's implementation — but it can surprise developers who expect code after an `.await` to always run. C#'s `CancellationToken` is explicit and cooperative — the developer controls exactly where cancellation is checked and what cleanup runs — but it is opt-in, meaning a method that ignores its token cannot be cancelled.

**The colored function problem.** Both languages suffer from function coloring: async functions cannot be called from synchronous functions without special handling (blocking the thread), and synchronous functions called from async context risk blocking the runtime's thread pool. In Rust, the distinction is invisible at the type level for blocking functions — there is no marker indicating "this function blocks" — which makes accidental misuse easy. In C#, the `async` keyword and `Task` return type make the distinction visible, but the fundamental incompatibility remains: `await` can only appear in `async` methods, and calling `.Result` or `.Wait()` on a `Task` from a synchronous method risks deadlocking if a `SynchronizationContext` is present.

Neither language has solved function coloring. Proposals for "keyword genericity" (abstracting over sync/async) exist in both ecosystems but remain unimplemented. In Rust, the "maybe(async)" concept has been discussed but faces limitations for intra-task concurrency patterns like `select!` and `join!`. In C#, the problem is partially mitigated by the ubiquity of the async model — modern .NET code is predominantly async, reducing the frequency of sync-to-async boundary crossings.

## 2. Thread Safety Markers

### Rust: Send and Sync

Rust provides two marker traits in `std::marker` that encode thread safety at the type level:

**`Send`** indicates that ownership of a value can be transferred to another thread. Precisely: if `T: Send`, then moving a `T` from one thread to another is safe (no undefined behavior, no data races). `Send` is an auto trait — the compiler implements it automatically for any type whose fields are all `Send`. It is also an `unsafe` trait — manual implementations require `unsafe impl` because the compiler cannot verify the safety invariant.

**`Sync`** indicates that a shared reference `&T` can be accessed from multiple threads simultaneously without data races. The formal definition is: `T: Sync` if and only if `&T: Send`. If it is safe to send a shared reference to another thread, then the referent can be safely shared. `Sync` is also an auto trait and an `unsafe` trait.

These two traits interact to form a complete thread-safety classification:

| Category | Send | Sync | Examples | Rationale |
|---|---|---|---|---|
| Freely shared | Yes | Yes | `i32`, `String`, `Arc<Mutex<T>>`, `Mutex<T>` (where `T: Send`) | Safe to move and to share references across threads. |
| Transfer only | Yes | No | `Cell<T>`, `RefCell<T>` | Safe to move between threads but not to share references, because interior mutation through `&self` uses non-atomic operations. |
| Thread-local | No | No | `Rc<T>`, raw pointers | Non-atomic reference counting; concurrent clone/drop causes undefined behavior. Cannot be moved to another thread or shared. |
| Immutably shareable only | No | Yes | `MutexGuard<T>` (on some platforms) | Rare category. The guard must be dropped on the same thread (POSIX pthreads requirement) so it is `!Send`, but sharing `&MutexGuard<T>` (which is read-only) across threads is safe if `T: Sync`. |

**Negative implementations.** Types can explicitly opt out: `impl !Send for Foo {}` (nightly) or by containing a `!Send` field (such as `PhantomData<*const ()>`). The standard library uses negative impls for `Rc`, raw pointers, and mutex guards.

**Compile-time enforcement.** The key insight is that `Send` and `Sync` are checked at function boundaries, not at usage sites. `std::thread::spawn` requires `F: Send + 'static` for the closure it receives. If the closure captures a `Rc<T>` (which is `!Send`), the program fails to compile. There is no runtime check, no overhead, and no possibility of a data race in safe code. The borrow checker's exclusivity rules (`&T` or `&mut T`, never both) combine with `Send`/`Sync` to make data races statically impossible: you cannot have concurrent mutable access to the same data, and you cannot share data across threads unless the type's `Sync` implementation guarantees safe concurrent read access.

### .NET: No Compile-Time Thread Safety Markers

The .NET type system has no equivalent of `Send` or `Sync`. Any reference can be shared between any threads at any time. Thread safety is a documentation convention, not a type-system property.

**Thread and ThreadPool.** `System.Threading.Thread` represents an OS thread. `ThreadPool.QueueUserWorkItem` schedules work on a managed thread pool. `Task.Run` schedules a delegate on the thread pool and returns a `Task`. None of these APIs impose type-level constraints on the data the delegate captures — any object can be passed to any thread.

**SynchronizationContext as a scheduling mechanism.** `SynchronizationContext` provides a way to marshal work to a specific thread (typically the UI thread) but does not provide safety guarantees. It is a scheduling mechanism, not a safety mechanism. Code that posts to the UI thread's `SynchronizationContext` is not prevented from also accessing the same object from a background thread.

**Thread safety as documentation.** The .NET documentation for each type specifies whether it is thread-safe for concurrent reads, concurrent writes, or both. `ConcurrentDictionary<K,V>` is documented as fully thread-safe. `List<T>` is documented as safe for concurrent reads but not concurrent writes. The compiler does not enforce these contracts. A developer who concurrently writes to a `List<T>` from two threads gets undefined behavior (corrupted state, `IndexOutOfRangeException`, torn reads) — but the code compiles and runs.

**.NET 9's `System.Threading.Lock`.** C# 13 introduced a dedicated `Lock` type that the `lock` statement recognizes. When the locked expression is of type `Lock`, the compiler generates `using (x.EnterScope())` instead of `Monitor.Enter`/`Monitor.Exit`. This improves performance (the `Lock` type uses a thin lock implementation) and type safety (the compiler warns if a `Lock` instance is cast to `object` and locked as a monitor). However, `Lock` is still a runtime mechanism — it prevents concurrent execution of critical sections, but the compiler does not verify that all accesses to shared state go through the lock.

### How Cobalt Could Bring Send/Sync to .NET

Cobalt could introduce marker traits analogous to `Send` and `Sync` as compile-time-only constructs — erased in the emitted IL but enforced during compilation. The compiler would:

1. Auto-derive `Send` and `Sync` for types whose fields are all `Send`/`Sync`, following Rust's auto-trait rules.
2. Require `T: Send` for values captured by closures passed to `Task.Run`, `Thread.Start`, or any API that transfers work to another thread.
3. Require `T: Sync` for values accessed through shared references across async tasks that may run on different threads.
4. Allow `unsafe impl Send` / `unsafe impl Sync` as escape hatches for types with manual synchronization that the compiler cannot verify.

Because these traits would be compile-time-only, they would impose no runtime cost and would not require changes to the CLR or existing .NET libraries. Interop with existing C# types would require either explicit `unsafe impl` declarations or a default policy (e.g., treat all unanalyzed .NET types as `Send + Sync` and accept the risk, or treat them as `!Send + !Sync` and require explicit annotation).

The challenge is that existing .NET types were not designed with these markers in mind. `List<T>` is `!Sync` (concurrent mutation is unsafe) but is routinely shared across threads by C# code that relies on external synchronization. Cobalt would need to either annotate the most common .NET types with correct `Send`/`Sync` bounds (a large, ongoing effort) or provide a mechanism for users to assert thread safety at the call site.

## 3. Data-Race Prevention

### Definitions

A **data race** occurs when two or more threads concurrently access the same memory location, at least one access is a write, and there is no synchronization ordering the accesses. Data races cause undefined behavior in both C++ and Rust (though Rust's safe subset makes them impossible). The consequences include torn reads, corrupted state, and violations of type invariants.

A **race condition** is a broader concept: a bug where the program's correctness depends on the relative timing of concurrent operations. Race conditions are logic errors — for example, a check-then-act pattern where the condition becomes invalid between the check and the act. Race conditions are possible in every concurrent programming model, including Rust's. No type system prevents them because they are properties of application logic, not memory safety.

### Rust: Compile-Time Data-Race Prevention

Rust's guarantee is precise: **data races are impossible in safe code.** This follows from two rules:

1. **Borrow checker exclusivity.** At any point in the program, a value is accessed either through one `&mut T` (exclusive, mutable) or through one or more `&T` (shared, immutable), never both. This eliminates concurrent read-write and write-write conflicts within a single thread and, combined with `Send`/`Sync`, across threads.

2. **Send/Sync enforcement.** Values cannot cross thread boundaries unless their type permits it. Shared access across threads requires `T: Sync`, which is only auto-implemented for types where `&T` is safe to access concurrently — primitives, atomics, and types protected by synchronization (like `Mutex<T>`, which provides `&T` access only after acquiring the lock).

Interior mutability types (`Cell<T>`, `RefCell<T>`, `Mutex<T>`, `RwLock<T>`, atomics) allow mutation through `&self` but do so safely: `Cell<T>` is `!Sync` (cannot be shared across threads), `RefCell<T>` is `!Sync` (same reason — its runtime borrow checking is not thread-safe), `Mutex<T>` is `Sync` (because it uses OS-level locking to serialize access), and atomics are `Sync` (because they use hardware-level atomic operations).

The result is that any program that compiles in safe Rust is free from data races. Race conditions, deadlocks, and other concurrency bugs remain possible — but memory corruption due to unsynchronized concurrent access cannot occur.

**Unsafe Rust.** The `unsafe` keyword allows code to bypass borrow checker and `Send`/`Sync` constraints. Unsafe code can create data races if it violates the invariants that safe code relies on. The `unsafe` boundary is narrow and auditable — it marks exactly where the compiler's guarantees end and the programmer's responsibility begins.

### C#/.NET: Runtime-Based Synchronization

.NET provides no compile-time data-race prevention. Thread safety is achieved through runtime synchronization primitives, correct usage of which is the developer's responsibility.

**lock and Monitor.** The `lock` statement provides mutual exclusion. It is syntactic sugar for `Monitor.Enter`/`Monitor.Exit` (or, in C# 13, `Lock.EnterScope()`). The lock ensures that only one thread executes the critical section at a time. The compiler does not verify that all accesses to the protected data go through the lock — it is possible to access the data outside the lock without any compiler diagnostic.

**Mutex and SemaphoreSlim.** `System.Threading.Mutex` is a kernel-level mutex that works across processes. `SemaphoreSlim` is a lightweight semaphore that limits the number of threads that can enter a critical section concurrently. Both are runtime primitives with no compile-time verification.

**ReaderWriterLockSlim.** Allows multiple concurrent readers or a single writer, analogous to Rust's `RwLock`. Again, no compile-time enforcement — the developer must ensure that all read accesses acquire a read lock and all write accesses acquire a write lock.

**Interlocked.** Provides atomic read-modify-write operations: `Interlocked.Increment`, `Interlocked.CompareExchange`, `Interlocked.Exchange`. These map to hardware atomic instructions and are the lowest-level synchronization primitive in .NET. They are correct by construction (the operation itself is atomic) but protect only individual operations — sequences of `Interlocked` calls are not atomic as a group.

**volatile.** The `volatile` keyword prevents the compiler and JIT from reordering reads and writes to a field. It does not provide atomicity — a `volatile long` on a 32-bit platform can still be torn. `volatile` ensures acquire semantics on reads and release semantics on writes, preventing stale cached values. `Thread.MemoryBarrier()` provides a full fence.

**Memory model.** The ECMA CLI specification defines a relatively weak memory model. The runtime guarantees:
- Reads and writes to variables of reference types and most primitive types (up to the platform's native word size) are atomic.
- Volatile reads have acquire semantics; volatile writes have release semantics.
- The JIT and hardware may reorder non-volatile reads and writes for performance.

On x86/x64 (which has a strong Total Store Order model), the hardware provides stronger guarantees than the ECMA spec requires — stores are not reordered with respect to other stores. On ARM (which .NET now targets for mobile and Apple Silicon), the hardware provides weaker ordering, and the difference between "works on x64" and "works per spec" becomes observable. Correct cross-platform .NET code must use `volatile`, `Interlocked`, or explicit memory barriers rather than relying on x86's strong ordering.

### Concurrent Collections

.NET provides a set of thread-safe collection types in `System.Collections.Concurrent`:

**`ConcurrentDictionary<TKey, TValue>`** — a thread-safe dictionary using fine-grained locking (striped locks). Individual operations (`TryAdd`, `TryGetValue`, `GetOrAdd`, `AddOrUpdate`) are atomic, but sequences of operations are not. The `GetOrAdd` overload that takes a factory delegate may invoke the factory multiple times if multiple threads race to add the same key — only one result is stored, but the factory has side effects.

**`ConcurrentQueue<T>`** and **`ConcurrentStack<T>`** — lock-free FIFO and LIFO collections implemented using `Interlocked` operations. They avoid kernel-mode transitions entirely, making them efficient for high-throughput producer-consumer scenarios.

**`ConcurrentBag<T>`** — an unordered thread-safe collection optimized for scenarios where the same thread produces and consumes items. It uses thread-local storage to minimize contention.

**`BlockingCollection<T>`** — a wrapper over `IProducerConsumerCollection<T>` (typically a `ConcurrentQueue<T>`) that adds bounding and blocking. `Take()` blocks when the collection is empty; `Add()` blocks when the collection is full (if bounded). It is the .NET equivalent of a bounded blocking channel.

**`Channel<T>`** (in `System.Threading.Channels`) — a modern producer-consumer primitive designed for async code. Channels come in bounded and unbounded variants. Writers use `WriteAsync` or `TryWrite`; readers use `ReadAsync`, `TryRead`, or `ReadAllAsync` (which returns `IAsyncEnumerable<T>`). Bounded channels support configurable full-mode behavior: wait, drop newest, drop oldest, or drop the write. Channels are the idiomatic way to implement producer-consumer patterns in modern async .NET code.

**`System.Collections.Immutable`** — immutable collection types (`ImmutableList<T>`, `ImmutableDictionary<TKey, TValue>`, `ImmutableArray<T>`, etc.) that are inherently thread-safe because they cannot be mutated. Operations like `Add` return a new collection sharing structure with the original. They are safe to share across threads without synchronization, at the cost of allocation on each "mutation."

## 4. Concurrency Patterns

### Rust

**Channels (std::sync::mpsc).** The standard library provides a multi-producer, single-consumer channel via `std::sync::mpsc::channel()` (unbounded) and `sync_channel(capacity)` (bounded). `Sender<T>` is `Clone` (multi-producer); `Receiver<T>` is not (single-consumer). Values sent through the channel transfer ownership — `T: Send` is required. The channel disconnects (returns `Err`) when all senders or the receiver are dropped.

The standard library's `mpsc` channels are limited to single-consumer and do not support async. The **crossbeam** crate provides multi-producer, multi-consumer channels with better performance characteristics. For async code, **tokio::sync::mpsc**, **tokio::sync::broadcast**, **tokio::sync::watch**, and **tokio::sync::oneshot** provide async-aware channels that integrate with the tokio runtime's waker mechanism.

**Scoped threads (std::thread::scope).** Stabilized in Rust 1.63, `std::thread::scope` provides structured concurrency for OS threads. Within a scope, spawned threads may borrow non-`'static` data from the parent stack frame. The scope guarantees that all spawned threads are joined before the scope exits, making borrowing sound — the compiler knows the parent data outlives the threads. This eliminates the need for `Arc` cloning in many parallel computation patterns.

```rust
let mut data = vec![1, 2, 3];
std::thread::scope(|s| {
    s.spawn(|| println!("{:?}", &data));  // borrows data — no Arc needed
});
data.push(4);  // safe — all threads are joined
```

Scoped threads exemplify structured concurrency: concurrent operations have explicit, lexical scope boundaries; the parent waits for all children before proceeding; panic propagation is automatic.

**Rayon (data parallelism).** The rayon crate provides parallel iterators via `par_iter()`, converting sequential `Iterator` chains into parallel execution using work-stealing scheduling. `par_iter().map(...).filter(...).collect()` parallelizes each stage across the available CPU cores. Rayon also provides `join(|| a, || b)` for fork-join parallelism and `scope(|s| { s.spawn(|| ...) })` for dynamic task spawning. All of these guarantee data-race freedom through the borrow checker — closures passed to rayon must satisfy `Send` bounds.

**Actor-like patterns.** Rust has no built-in actor framework, but the combination of channels and `tokio::spawn` produces actor-like patterns: a long-lived task receives messages through a channel, processes them sequentially, and sends responses through another channel. The `actix` and `xtra` crates formalize this pattern. Ownership transfer through channels means the actor owns its state exclusively — no locking needed.

### C#/.NET

**Task Parallel Library (TPL).** The TPL provides high-level constructs for parallelism:
- `Parallel.For` and `Parallel.ForEach` — partition a range or collection and process partitions concurrently on thread pool threads. The runtime dynamically adjusts the degree of parallelism based on available cores and thread pool utilization.
- `Parallel.ForEachAsync` (added in .NET 6) — an async-aware variant that accepts an `async` delegate and limits concurrency via `ParallelOptions.MaxDegreeOfParallelism`.
- `Task.WhenAll` and `Task.WhenAny` — combinators for concurrent async operations. `WhenAll` completes when all input tasks complete; `WhenAny` completes when the first input task completes.

**PLINQ (Parallel LINQ).** `AsParallel()` converts a LINQ query into a parallel query. The runtime partitions the data source, executes the query operators in parallel, and merges the results. PLINQ supports `CancellationToken` via `WithCancellation()` and degree-of-parallelism control via `WithDegreeOfParallelism()`.

**System.Threading.Channels.** As described in section 3, `Channel<T>` provides bounded and unbounded async producer-consumer channels. Channels separate the `ChannelWriter<T>` (producer) from the `ChannelReader<T>` (consumer), allowing them to be distributed to different components. `ChannelReader<T>.ReadAllAsync()` returns `IAsyncEnumerable<T>`, integrating channels with `await foreach`.

**TPL Dataflow.** The `System.Threading.Tasks.Dataflow` namespace provides a set of building blocks for data-processing pipelines: `BufferBlock<T>`, `TransformBlock<TIn, TOut>`, `ActionBlock<T>`, `BatchBlock<T>`, and others. Blocks are linked into a pipeline; data flows from producers through transform stages to consumers. Each block processes items concurrently (configurable degree of parallelism) and manages its own internal buffer. Dataflow is more structured than raw channels but less widely adopted.

**Actor frameworks.** .NET has third-party actor frameworks:
- **Microsoft Orleans** — a virtual actor ("grain") framework designed for distributed systems. Grains are single-threaded by default (turn-based concurrency), eliminating the need for locks within a grain. Orleans manages grain activation, deactivation, and distribution across a cluster.
- **Akka.NET** — a port of the Akka actor framework from the JVM. Provides hierarchical supervision, message routing, and distributed actors.
- **Proto.Actor** — a lightweight actor framework that supports .NET and Go.

**Structured concurrency.** .NET does not currently have a built-in structured concurrency primitive equivalent to Rust's `std::thread::scope`. Tasks spawned with `Task.Run` are unstructured — they can outlive their creating scope, and there is no lexical guarantee that they will be awaited. Stephen Toub has discussed structured concurrency proposals for .NET, but as of .NET 9, no language-level or library-level support has shipped. The closest approximation is `Task.WhenAll` inside a method: `await Task.WhenAll(task1, task2, task3)` waits for all tasks before continuing, but the tasks themselves were already started eagerly and could have been observed (or leaked) before the `WhenAll`.

## 5. Implications for Cobalt

### Using .NET's Task Infrastructure with Rust-Style Safety

Cobalt's async model should build on .NET's existing `Task` and `ValueTask` infrastructure rather than reinventing it. The .NET thread pool, `Task` scheduler, and `IAsyncStateMachine` transformation are mature, optimized, and interoperable with the entire .NET ecosystem. Departing from them would sacrifice interop with every async .NET library.

The value Cobalt adds is compile-time safety layered on top of this infrastructure:

**Send bounds on Task.Run.** When a Cobalt closure is passed to `Task.Run`, the compiler should verify that all captured variables satisfy `Send` — they can be safely transferred to a thread pool thread. A closure capturing a `Rc<T>` equivalent (if Cobalt has one) or a non-thread-safe reference would fail to compile. This is precisely the guarantee that Rust's `tokio::spawn` and `std::thread::spawn` provide.

**Sync bounds on shared async state.** If an async task holds a shared reference (Cobalt's equivalent of `&T`) to data that might be accessed concurrently by another task on a different thread, the compiler should require `T: Sync`. This prevents sharing non-thread-safe types (types with non-atomic interior mutability) across concurrent tasks.

**Borrow checker across await points.** The borrow checker should track references across `.await` points, enforcing that borrowed data outlives the suspension. This is analogous to Rust's requirement that futures hold only valid borrows. On .NET, this interacts with the GC: managed references remain valid as long as the GC keeps the referent alive, but Cobalt-owned values with deterministic lifetimes need careful handling. If an owned value is dropped while a task still holds a borrow to it, the compiler should reject the program.

### Cancellation: CancellationToken vs Drop-Based

Cobalt should use `CancellationToken` for interoperability with the .NET ecosystem. Every async .NET API accepts `CancellationToken`; ignoring this convention would make Cobalt async methods awkward to consume from C# and vice versa.

However, Cobalt could layer drop-based cleanup on top: when a Cobalt future (backed by a `Task`) is dropped (ownership ends), the compiler could automatically trigger cancellation on an internally-managed `CancellationTokenSource` and dispose of resources held by the state machine. This provides the ergonomic benefit of Rust's drop-based cancellation (resources are cleaned up when the future is abandoned) while maintaining compatibility with .NET's cooperative cancellation model.

The key difference from Rust is that .NET cancellation is not immediate — the cancelled task's code may continue executing until it checks the token. Cobalt's documentation should make this clear: dropping a Cobalt future requests cancellation but does not guarantee that the underlying work stops instantly. For resources managed by RAII-style `Drop`/`Dispose`, cleanup occurs when the state machine observes the cancellation and unwinds.

### The Colored Function Problem

Cobalt cannot fully solve the colored function problem while targeting .NET, because the distinction between synchronous and asynchronous execution is fundamental to the .NET runtime model — `Task`-returning methods have different calling conventions, different state machine representations, and different scheduling behavior than synchronous methods.

Cobalt can mitigate the problem in several ways:

**Implicit async propagation.** If Cobalt defaults to async execution for I/O-bound operations (similar to how some functional languages handle effects), the distinction becomes less visible. However, this conflicts with .NET interop expectations — C# callers expect to see `Task<T>` return types for async methods.

**Sync-from-async without deadlock.** Cobalt could provide a safe `block_on` primitive that detects and prevents deadlocks caused by blocking on a `Task` when a `SynchronizationContext` is present. This would not eliminate the color distinction but would make crossing the boundary safer.

**Effect-system integration.** A more ambitious approach: Cobalt could treat async as an effect, using an effect-system-like mechanism to allow generic code that is polymorphic over sync/async. This is an active area of PL research (algebraic effects, capabilities) and would be novel for a .NET language. The implementation complexity is significant, and the interaction with .NET's concrete `Task` type is nontrivial.

The pragmatic recommendation is to accept the colored function distinction (as both C# and Rust do) and focus Cobalt's innovation on safety — `Send`/`Sync` bounds, borrow checking across `await`, and deterministic resource cleanup — rather than attempting to unify sync and async execution models.

### What Compile-Time Thread Safety Guarantees Are Achievable on .NET

The strongest achievable guarantee on .NET is: **data races involving Cobalt-managed values are impossible in safe Cobalt code.** This is the same guarantee Rust provides, implemented through the same mechanisms (borrow checking + `Send`/`Sync` marker traits), but applied to values managed by Cobalt's ownership system.

This guarantee has boundaries:

**Cobalt-to-Cobalt code.** Full data-race prevention. The borrow checker ensures exclusivity. `Send`/`Sync` traits ensure thread-safe sharing. No runtime overhead.

**Cobalt-to-C# interop.** At the interop boundary, Cobalt cannot verify that C# code respects thread safety. Values passed to C# code cross into unchecked territory. Cobalt should provide `unsafe` interop blocks for this case, making the developer responsible for thread safety at the boundary — analogous to Rust's `unsafe` blocks around FFI calls.

**GC-managed objects.** Objects on the managed heap are subject to GC relocation (handled by the runtime — references are updated) but also subject to concurrent access from any thread holding a reference. Cobalt's ownership system can track which threads hold references to a Cobalt-managed object, but it cannot control access from C# code that holds a reference to the same object. The interop boundary is where safety guarantees degrade.

**Lock safety.** Cobalt could go beyond `Send`/`Sync` and provide a `Mutex<T>` type that, like Rust's, returns a guard that provides exclusive access to the inner value. The guard would implement `Drop` (auto-release the lock when the guard goes out of scope) and `Deref`/`DerefMut` (provide access to the protected value only through the guard). This would make it impossible to access the protected data without holding the lock — a guarantee that .NET's `lock` statement does not provide, because `lock` protects a code block, not a value.

**Async-specific guarantees.** Cobalt could enforce that values borrowed across `.await` points satisfy `Send` (since the continuation may resume on a different thread) and that mutable references are not held across `.await` points unless the referenced value is pinned or `Send`. This would catch a class of bugs that are currently runtime failures in C# (e.g., holding a non-thread-safe lock guard across an `await`, which the C# compiler warns about for `lock` but cannot enforce for other synchronization primitives).

In summary, Cobalt can provide Rust-level data-race prevention for code written in Cobalt, degrading to C#-level safety (runtime enforcement, programmer responsibility) at interop boundaries. The `unsafe` keyword marks exactly where this degradation occurs, making the safety boundary explicit and auditable.
